# ADR-001: Monorepo Structure and Adapter Pattern

## Status
Accepted

## Context
We are choosing the foundational repository structure and software design pattern for the JARVIS AI assistant. The project involves multiple technologies: a Next.js frontend, an API backend, a Python execution environment for AI orchestration, and various shared definitions. Additionally, the system needs to be highly configurable with respect to LLMs, Vector Databases, Memory storage, and other services.

## Decision
1. **Monorepo Structure**: We will use a `pnpm` workspace monorepo. This allows us to share TypeScript interfaces across the web and API packages, manage dependencies efficiently, and coordinate work across languages (including Python via a dedicated package directory containing a virtual environment).
2. **Adapter Pattern**: To ensure JARVIS remains highly pluggable and avoids vendor lock-in, all core integrations (LLMs, Memory Stores, Vector Stores) must implement defined adapter interfaces located in the `@jarvis/shared` package. Implementations of these adapters will exist in the respective application code but adhere to the agreed contracts.

## Consequences
- Requires using `pnpm` workspaces.
- Python environment lives alongside JS/TS but dependency management is separate.
- High degree of modularity and testability.
