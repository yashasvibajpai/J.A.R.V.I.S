# ADR-002: Deployment Architecture

## Status
Accepted

## Context
As JARVIS moves from a local development environment (Phase 1) towards a production-ready always-on application, we need a deployment strategy that minimizes costs (preferably free tier) while ensuring reliability, ease of updates, and secure communication between the frontend and backend.

The architecture consists of:
1. **Frontend PWA:** Next.js React application (`packages/web`).
2. **Backend API:** Node.js Express server (`packages/api`).
3. **Data Stores:** SQLite databases for Memory and Profile (currently local, moving to cloud/Supabase eventually).
4. **LLM Providers:** External APIs (Anthropic, OpenAI) and optionally a local node running Ollama.

## Decisions

### 1. Frontend: Vercel
- **Why:** Vercel provides zero-configuration deployment for Next.js applications with a generous free tier. It offers edge caching, built-in CI/CD via GitHub, and optimal performance for server-side rendered or statically exported Next.js pages.
- **How:** A `vercel.json` and root-level build commands map the Vercel builder to our `packages/web` workspace project. 

### 2. Backend: Railway (or Supabase Edge/Fly.io)
- **Why:** We need a Node.js runtime that can execute our Express server continuously, expose an HTTP port, and securely store environment variables (API keys). A Dockerized build provides the most portable and robust deployment model. Railway offers an excellent free/hobby tier with native Docker support.
- **How:** A `Dockerfile.api` describes the multi-stage build for the Express application. `railway.toml` acts as the service configuration pointing to the Dockerfile.

### 3. CI/CD: GitHub Actions
- **Why:** We need to ensure that no code goes to production if it breaks core types across the monorepo packages. 
- **How:** A GitHub Actions workflow (`ci.yml`) runs on every push to `main` and Pull Requests. It executes `pnpm build` and `pnpm lint` across the monorepo to guarantee type safety and code quality before Vercel or Railway fetch the updates.

### 4. Zero-CORS Development via Next.js Proxy
- **Why:** Running the backend and frontend on different ports locally or domains in production often introduces CORS overhead.
- **How:** We use Next.js rewrites (`next.config.ts`) to proxy requests to `/api/*` to the backend. In Vercel, this can eventually be handled by mapping routes if both are hosted together, or simply keeping the rewrite to point to the Railway URL.

## Consequences
- **Positive:** Deployment is automated and decoupled. The frontend benefits from global CDN caching while the backend is securely isolated. Use of Docker for the API ensures we are not locked into any single cloud provider.
- **Negative:** For users wanting a pure "local only" JARVIS deployment (e.g. running perfectly offline on a laptop with Ollama), cloud deployment steps are irrelevant. We will document both "cloud" and "local" deploy paths. Since Memory and Profiles use SQLite, deploying the API via Docker needs persistent volume mounts (which Railway supports) so memory isn't wiped on container restarts.

## Future Path
When implementing Phase 2 and 3, SQLite might be migrated to a hosted Postgres (Supabase) to eliminate the need for persistent disk volumes on the API container, making the API entirely stateless.
